# v0.4.2a Non-Circular Validation Pipeline

**Date archived:** 2026-07-10
**Source version:** v0.4.2a
**Source script:** `v0.4.2a/non_circular_validation.js`
**Source results:** `v0.4.2a/NON_CIRCULAR_RESULTS.json`

## Method

Test INV-005 and INV-006 WITHOUT pre-filtering on header magic. Accept ANY marker match (no edgeCount, vertexCount, or header filtering). Read bytes at computed Block 1 and Block 2 positions and test headers only after reading.

This is the only way to falsify INV-005 and INV-006. It produces many false positives (non-face marker matches), which is expected.

## Corpus

Same 8 openswx files. Total candidates (no filter): 4,936.

## Results

**INV-005 (B1 header [4,8,2,N]):**
- PASS: 1234 (candidates where B1 header is readable AND valid)
- FAIL: 3702 (candidates where B1 header is readable but NOT [4,8,2,N])
- All failures are clearly false positives: edgeCount values are invalid floats (e.g., 3164048720), vertexCount=0, header bytes are float data or other non-structure.

**INV-006 (B2 header [4,8,2,M]):**
- PASS: 1234
- FAIL: 0
- Every candidate with a readable B2 position has [4,8,2,M].

## Limitations

These results strengthen confidence in INV-005 and INV-006 but do not fully remove all assumptions of the extraction pipeline:

1. **Marker selection**: Faces are still located by the [12,100,2,vc] gap marker pattern. Any face not preceded by this marker would be missed.
2. **Normal-offset layout assumption**: The pipeline assumes normals follow immediately after the 16-byte gap. This is empirically validated but remains an assumption.
3. **Vertex-count bounds**: The vc limit (6000) bounds the search. Extremely large faces could be excluded.
4. **Float validity filters**: Position validation discards candidates with non-finite or extreme float values.
5. **Block ordering**: The pipeline assumes positions → gap → normals → Block 1 → Block 2 in strict linear order.
