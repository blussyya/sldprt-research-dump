## Method

Search for all [4,8,2,N] patterns within ±256 bytes of each face's true B1 position. Analyze their properties and correlations.

### Formula

The delta follows the formula:
```
delta = -48 - 24*vc - 4*N
```

This means the alternative is always at position `mp - 16 - 4*N` (16 bytes before edgeCount for N=1, 20 bytes for N=2).

### Structure Analysis

**N=1 alternatives:**
- Header: [4, 8, 2, 1] at mp-20
- Body: [edgeCount] at mp-4 (1 word)

**N=2 alternatives:**
- Header: [4, 8, 2, 2] at mp-24
- Body: [prev_edgeCount, edgeCount] at mp-8, mp-4 (2 words)

## Observations

1. 332 out of 1,234 faces (26.9%) have alternative [4,8,2,N] headers
2. Every face has at most ONE alternative (no face has multiple)
3. All alternatives are BEFORE the true B1 (100%)
4. N values are only 1 or 2 (no other values observed)
5. Delta follows formula: delta = -48 - 24*vc - 4*N
6. Alternative is always at mp - 16 - 4*N
7. N=1 body: second word equals current face's edgeCount
8. N=2 body: second word equals current face's edgeCount
9. HEADPHONE has zero alternatives (0 out of 62 faces)
10. PTC has highest alternative rate (102 out of 126 faces, 81.0%)

## Hypotheses

H1: The alternative [4,8,2,N] structure is a "face header" that appears before the face marker
H2: N represents the number of edgeCount values stored
H3: N=1 body: first word equals edgeCount (confidence: Medium)
H4: N=2 body: first word equals previous face's edgeCount (confidence: Low)
H5: The structure is related to face adjacency or topology (confidence: Low)
H6: HEADPHONE faces have a different structure that doesn't include this header (confidence: Low)

## Confidence

High: The structure is real and follows a deterministic formula
Medium: N=1 body contains edgeCount
Low: N=2 body contains previous edgeCount; semantic meaning

## Unknowns

1. Why do some faces have this header and others do not?
2. What determines whether N=1 or N=2?
3. Why does HEADPHONE have zero alternatives?
4. Is this structure specific to openswx format or present in other SolidWorks versions?
5. Does this structure affect geometry parsing or is it metadata?

## Critical Review Added 2026-07-16

### Discrepancy: EXP-019 vs EXP-021 Counts

EXP-019 reported 525 faces with alternative [4,8,2,N] headers. EXP-021 reports 332. The 193-face difference is explained by EXP-019 counting B2 header positions as "alternatives" (since its search included the B2 area without recognizing B2 headers). EXP-021 correctly excludes B2 positions via the `isB2` filter. This discrepancy is not mentioned in the main report.

**Taking this into account:** EXP-019's original falsification of H4 - "No alternative B1 positions" — is weaker than stated. 193 of 525 "alternatives" were actually B2 headers, not B1 alternatives. The real count of non-B2 alternatives is 332, not 525.

The remaining positive-delta alternatives from EXP-019 (delta=120, 128 with N=1, N=2, N=3) were found by EXP-019 but NOT by EXP-021. These are within the B1 or post-B2 area. EXP-021's wider range should have found them. This is an unresolved discrepancy.

### What Was Not Tested

1. **The alternative has not been validated as a parseable face header.** If we treat the alternative as the true face container start, does reading edgeCount, computing vertexCount, and locating B1/B2 from this offset produce valid results? This test would determine whether the alternative is a genuine B1 that defines a valid face boundary, or a coincidental pattern.

2. **The "prev_edgeCount" claim for N=2 has not been verified.** The evidence asserts that the second body value at mp-8 is the previous face's edgeCount. This can only be verified by traversing faces in DisplayLists order and checking whether mp-8 of the current face equals the edgeCount at mp-4 of the previous face in sequence. This test was not performed.

3. **Non-accepted faces were never examined.** The experiment only checks faces that survive the extraction pipeline (ec in [1,500], vc in [3,6000], valid floats, exact gap match, valid B1/B2 headers). Faces rejected at any filter stage — which could include genuine faces with a different layout — were not searched for alternative headers. If a face type exists that doesn't match the current extraction criteria BUT has a [4,8,2,N] header at mp-16-4*N, it would be invisible to this experiment.

4. **The alternative structure was not searched for in non-face regions of DisplayLists.** The experiment only looks within ±256 bytes of accepted B1 positions. If this structure also appears elsewhere in the stream (e.g., between faces, in non-face sections), the conclusion that it is "a face header" would need revision.

5. **HEADPHONE's zero-count is unexplained but potentially damaging.** HEADPHONE has 62 faces, all of which pass the extraction pipeline and all invariants. Yet zero have the alternative header. Possible explanations:
   - HEADPHONE uses a different SolidWorks version or exporter
   - The structure is optional metadata, not required for geometry
   - HEADPHONE's faces use a different container format

   If (a), the structure is version-specific, not universal. If (b), the structure is metadata, not part of the face grammar. If (c), multiple face container formats exist. The current evidence cannot distinguish these.

### Hidden Assumptions

1. **The alternative is a meaningful structure, not a coincidence.** The deterministic formula and consistent body content strongly suggest meaning, but this has not been proven by parsing the face from the alternative position.

2. **The search range (±256 bytes from B1) is sufficient.** If alternatives exist at larger offsets, they are missed.

3. **u32-aligned search is sufficient.** If the header could appear at non-4-byte-aligned positions, it is missed.

4. **Only [4,8,2,N] patterns are valid container headers.** Alternative container formats (different magic bytes) would be missed.

5. **The extraction pipeline finds all genuine faces.** This is the foundational assumption of both EXP-018 and EXP-021. If the pipeline rejects a genuine face type, its alternative header (and the face itself) is invisible.

### Overclaimed Conclusions

**Claim:** "The structure is related to edgeCount storage."

**Reality:** The body contains values that match the known edgeCount field. For N=1, it's a single edgeCount. For N=2, the experiment asserts "previous face's edgeCount" but never verifies this by cross-checking adjacent faces. The claim is plausible but unverified.

**Claim:** "The structure is real and follows a deterministic formula" — Confidence: High.

**Assessment:** This is sound. The pattern is real (332 instances across 7 of 8 files), and the delta formula predicts all observed positions. This conclusion is warranted.

**Claim:** "N represents the number of edgeCount values stored" — Confidence: Medium.

**Assessment:** Reasonable but incomplete. The body contains N u32 values that end at mp-4 (the known edgeCount position). The claim that these are edgeCount values specifically (rather than some other per-face metric) relies on the last body word matching the already-known edgeCount field at mp-4. For N=1, the match is exact, so the body word IS edgeCount. For N=2, the claim that the first body word is "prev_edgeCount" is unverified.

### What This Changes About The Face Layout Model

The face layout should be updated to reflect an outer container:

```
BEFORE (previously known layout):
  mp-4: edgeCount
  mp+0: [12, 100, 2, vertexCount]

AFTER (with EXP-021 alternative header, N=1 case):
  mp-20: [4, 8, 2, 1]  — outer container header
  mp-4:  edgeCount     — container body (1 word)
  mp+0:  [12, 100, 2, vertexCount]  — face marker (previously considered "face start")

For faces WITHOUT the alternative header:
  mp-4: edgeCount
  mp+0: [12, 100, 2, vertexCount]
```

This suggests the face is wrapped in an optional [4,8,2,N] container that duplicates or augments the edgeCount. The container is present for 26.9% of faces and absent for 73.1%.

The determinant of presence/absence is unknown. HEADPHONE (0/62) and PTC (102/126, 81.0%) are the extremes, suggesting a non-random distribution that may correlate with SolidWorks version, model type, or export settings.

## Critical Review Update — 2026-07-16 (v0.4.5)

### Falsified: N=2 "prev_edgeCount" Claim

**Claim from EXP-021 evidence:** "N=2 alternatives contain [prev_edgeCount, edgeCount] as body" (line 68); body at mp-8 = previous face's edgeCount.

**Test:** Traverse faces in DisplayLists order. For each face with an N=2 alternative ([4,8,2,2] at mp-24), read body[0] at mp-8 and compare to the previous face's edgeCount at mp-4.

**Script:** `v0.4.3/docs/research/exp021_prev_edgecount_falsification.js`

**Results across 8-file corpus:**

| File | N=2 Faces | Verified | Failed | Body0 Distribution |
|------|-----------|----------|--------|-------------------|
| BOTTOM | 4 | 0 | 4 | {3: 4} |
| TOP | 4 | 0 | 4 | {3: 4} |
| GEAR | 25 | 4 | 20 | {3: 14, 4: 2, 5: 8, 33: 1} |
| DEKOR | 204 | 3 | 201 | {3: 168, 5: 21, 7: 5, 9: 7, 11: 1, 13: 2} |
| HEADPHONE | 7 | 0 | 7 | {3: 7} |
| DISTRIBUTOR | 11 | 0 | 11 | {3: 9, 21: 1, 25: 1} |
| POCKET | 42 | 0 | 42 | {3: 42} |
| PTC | 3 | 0 | 3 | {3: 2, 4: 1} |
| **Total** | **300** | **7** | **292** | **3: 241, 5: 21, 4: 3, 7: 5, 9: 7, 11: 1, 13: 2, 21: 1, 25: 1, 33: 1** |

**Verdict: FALSIFIED.** Only 7 of 292 tested N=2 faces (2.4%) verify the prev_edgeCount claim. 97.6% fail. The value at mp-8 is consistently 3 (80.3% of cases) regardless of the previous face's actual edgeCount.

**What mp-8 IS NOT:**
- Not the previous face's edgeCount (falsified)
- Not the face's own vertexCount (checked: 0/292 match)
- Not correlated with face index position

**What mp-8 IS:**
- Predominantly 3 (241/300 cases)
- Varies across files: some files have 100% `3` (BOTTOM, TOP, HEADPHONE, POCKET); others have mixed values (GEAR, DEKOR)
- Non-3 values (4,5,7,9,11,13,21,25,33) occur in specific files but don't correlate with ec, vc, or face adjacency

**SEMANTICS UNKNOWN.** The second body word of N=2 alternatives is unidentified.

### Discrepancy: EXP-021 vs EXP-023 Alternative Counts

Excerpt from `knowledge/evidence/2026-07-16_v0.4.4-EXP023.md`:

| Metric | EXP-021 | EXP-023 |
|--------|---------|---------|
| Total faces | 1,234 | 1,172 |
| Faces with alternatives | 332 (26.9%) | 661 (56.4%) |
| N=1 | 240 | 368 |
| N=2 | 92 | 293 |

**Discrepancy: EXP-023 finds 2× the alternatives of EXP-021.** Root cause: EXP-021 restricted its search to ±256 bytes from B1 of each face. EXP-023 checks fp-20/fp-24 for every face regardless of B1 proximity. Since some alternatives fall outside ±256 of B1, EXP-021 misses them.

**Implication:** The ±256 constraint was an arbitrary limit that introduced a blind spot. EXP-021's count of 332 alternatives is an underestimate. EXP-023's count of 661 (56.4% of faces) is the better estimate.

**Remaining concern:** EXP-023's detection at fp-20/fp-24 has no structural constraint beyond the [4,8,2,N] pattern itself. Some detections could be false positives if the bytes at these positions coincidentally match the pattern. However, the deterministic relationship between altPos = mp - 16 - 4*N makes false positives unlikely — random data would not consistently position [4,8,2,N] at exactly mp-20/mp-24 with N matching the expected body size.

### Recommended Next Experiments

1. Determine actual meaning of N=2 body word at mp-8 (not prev_edgeCount). Scan for correlation with per-face data at offset from face start, not from alt start.

2. For N=1 alternatives, parse the face treating the alternative as the true container start. Check if edgeCount, vertexCount, and B1/B2 positions produce valid invariants from this new reference point.

3. Search the entire DisplayLists stream for [4,8,2,N] patterns to determine whether this is a stream-wide container format or face-specific.

4. Examine HEADPHONE's DisplayLists structure to determine why it lacks alternatives — compare byte-level patterns around face entries to files that do have alternatives.

5. Re-run EXP-019 H2/H3/H5 with genuinely independent definitions (not asserting what they test). These hypotheses were tautologies in v0.4.3 — they defined terms circularly and cannot count as "surviving" tests.

6. Investigate the `3` at mp-8: is it a type tag, fixed offset, or metadata flag? Compare with files from different SolidWorks versions.
