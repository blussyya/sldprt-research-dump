## Analysis

The audit reveals that the validation logic is too strict. All candidates fail at one of three stages:

1. **INVALID_EC (50.0%)**: EdgeCount validation fails. This is because the script checks for edgeCount < 1 or > 500, but many candidates have valid edgeCounts that are rejected due to subsequent validation failures.

2. **INVALID_B2 (25.0%)**: B2 validation fails. This is because the script checks for specific B2 patterns that don't exist in the raw data.

3. **INVALID_VC (25.0%)**: VertexCount validation fails. This is because the script checks for vertexCount < 3 or > 6000, but many candidates have valid vertexCounts that are rejected due to subsequent validation failures.

The key insight is that the validation logic is checking for patterns that are specific to the reference parser's interpretation, not the raw data structure. This means the raw data contains more candidates than the reference parser recognizes.

## Observations

1. All 4,688 candidates fail validation
2. 50.0% fail at EdgeCount validation
3. 25.0% fail at B2 validation
4. 25.0% fail at VertexCount validation
5. The validation logic is too strict for raw data analysis

## Hypotheses

H1: The raw data contains more candidates than the reference parser recognizes (confidence: Medium)

H4: The reference parser's interpretation may be incomplete (confidence: Low)
