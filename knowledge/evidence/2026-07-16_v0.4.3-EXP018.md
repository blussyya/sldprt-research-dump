## Method

Scan DisplayLists for ALL `[12,0,0,0,100,0,0,0]` marker occurrences. Read raw bytes at computed positions without any pre-filtering. Classify candidates post-hoc.

## Key Observation

PARTIAL_GAP candidates ALL have:
- `edgeCount = 1`
- `vertexCount = 0`
- `gap = [12, 100, 2, 0]` — matches gap pattern exactly (since vc=0)
- `b1Header = [0, 0, 0, 0]` — all zeros, invalid

This is a clean separation:
- **FULL**: genuine faces with valid structure
- **PARTIAL_GAP**: false positives with vc=0 that happen to match the gap pattern
- **REJECTED**: false positives that don't match the gap pattern

## Facts

## Hypotheses

H1: The gap pattern [12,100,2,vc] is self-describing — the fourth word IS the vertexCount
H2: For false positives with vc=0, the gap "matches" because [12,100,2,0] matches [12,100,2,0]
H3: The B1 header [0,0,0,0] is invalid, which is why these are PARTIAL_GAP, not FULL

## Confidence

High that the current extraction model finds all genuine faces. The 1,234 FULL candidates match the previous parser output exactly.

## Remaining Assumptions

A1: Face located by [12,0,0,0,100,0,0,0] marker at faceStart+4
A2: edgeCount at faceStart+0 is u32 in [1,500]
A3: vertexCount at marker+12 is u32 in [3,6000]
A4: Positions are float32[vc*3] starting at marker+16
A5: Gap marker [12,100,2,vc] at positionsEnd
A6: Normals are float32[vc*3] immediately after gap (16 bytes)
A7: Block 1 at normalsEnd with header [4,8,2,N]
A8: Block 2 immediately after Block 1 with header [4,8,2,M]
A9: Block ordering: positions -> gap -> normals -> B1 -> B2
